# 全站文案温润化改版

## Problem Statement

Flipia 当前的页面文案偏向后台管理系统风格，措辞生硬、术语化（"配置""导入""禁用""抽取方案"等），缺乏生活温度。用户虽然能理解功能，但感觉像是在用一套工具，而不是一个有性格的产品。

## Solution

对全站 11 个页面的用户可见文案进行温润化改版：用日常生活口语替换技术化表达，统一为治愈、轻松的语调。核心原则：

- **去技术词**：配置、导入、禁用、管理 → 日常说法
- **缩短字数**：专有术语替换控制在 6 字以内
- **治愈语调**：错误不责备、完成用轻快语气、对话框像聊天
- **统一表达**：同义文案跨页一致（如"再试一下？"统一所有失败场景）

## User Stories

1. 作为 Flipia 用户，我希望首页不再是"开始抽取""一键翻牌"这种生硬按钮，而是像朋友一样问我"今天吃什么？"，让我在打开 App 的瞬间就感到轻松。
2. 作为 Flipia 用户，我希望抽取后的确认按钮不再是"就吃这个"，而是轻松的"就它了"，让决策变得愉快而非负担。
3. 作为 Flipia 用户，我希望菜品开关不再是"今日供应/今日停供"这种食堂味，而是"今天备着/今天不备"这种日常说法。
4. 作为 Flipia 用户，我希望错误提示不再是"加载失败，请重试"这种冷冰冰的句式，而是"出了点问题，再试一下？"这种不责备的说法。
5. 作为 Flipia 用户，我希望成功 toast 不再是"记录成功"，而是"记好啦"这种轻快的治愈感表达。
6. 作为 Flipia 用户，我希望菜品管理页的创建者标签不再是"大厨：xxx"，而是"出品人：xxx"，更有日系生活感。
7. 作为 Flipia 用户，我希望表单标签不再是"菜品名""烹饪描述"，而是"叫什么""做法 & 备注"，更口语化。
8. 作为 Flipia 用户，我希望侧边栏菜单项不再是"插件管理""分类配置"，而是"更多功能""分类整理"，去掉后台管理系统味。
9. 作为 Flipia 用户，我希望删除确认弹窗不再是"确认删除该条记录？删除后不可恢复"，而是"确定删掉？删了就找不回了"，直接且不吓人。
10. 作为 Flipia 用户，我希望新建厨房的步骤文案不再是"从已有厨房导入""选择源厨房"，而是"从别的厨房搬菜""从哪个厨房搬"，让我感觉在搬家而不是在操作数据库。
11. 作为 Flipia 用户，我希望厨房管理的解散弹窗不再是"解散后厨房及所有数据将被永久删除，不可恢复"，而是"关掉后所有菜和记录都会消失，找不回的"，说人话。
12. 作为 Flipia 用户，我希望邀请分享文案不再是"邀请你加入「xxx」"，而是"来「xxx」一起做饭吧"，有社交温度。
13. 作为 Flipia 用户，我希望能看懂校验错误——"分类名不能为空"变成"得有个名字"，"方案名已存在"变成"这个名字用过了"，不指责用户。
14. 作为 Flipia 用户，我希望全部页面的失败/错误场景都用"再试一下？"收尾，形成统一的治愈语调。
15. 作为 Flipia 用户，我希望历史页空状态不再是"暂无记录""抽取确认后记录将出现在这里"，而是"还没记录呢""抽完确认后，这里会记下来"。

## Implementation Decisions

### 改动范围

全站 11 个页面 + 共享库文案，分 4 层：

**第 1 层：WXML 静态文案** — 页面内的标签、按钮、占位符、弹窗文案
**第 2 层：TS toast/modal 文案** — `wx.showToast`、`wx.showModal`、`wx.showLoading` 内的 title/content
**第 3 层：lib/ 共享库文案** — `lib/category-manage.ts`、`lib/draw-config-manage.ts`、`lib/content-security.ts`、`lib/sanitize.ts` 等
**第 4 层：配置常量** — `config.ts` 中的默认名、`app.json` 中的 tab 标签

### 不改的项目

- tabBar 四个标签："首页""菜品""历史""我的" — 原已简洁
- 系统弹窗原生按钮："确认""取消"
- 品牌名 `"Flipia"`
- `"吃什么呢？交给 Flipia 吧～"`（首页空闲引导语）
- `"Flipia时刻"`（历史页品牌图文案）
- `"厨师长"`（厨房管理群主徽章）
- `"雨露均沾"`（默认方案名）
- `"我的厨房"`（默认厨房名）
- 预设分类名（荤菜、素菜、主食、汤）
- 预设菜品名（红烧肉等 20 道）

### 跨页面统一原则

- 所有 toast 失败场景统一为 `"没{动词}成，再试一下？"`（没记上、没存上、没删成、没搬成、没建成、没加成、没请成、没换成、没切过来 等）
- 通用兜底错误统一为 `"出了点问题"` + `"没成，再试一下？"`
- 校验错误统一为 `"得有个名字"` / `"这个名字用过了"`
- 完成 toast 统一为口语化 + 语气词（记好啦、建好啦、搬好啦、分类加好啦、图传好啦、换好啦、解锁啦）

### 底层逻辑不变

所有改动仅涉及展示文案字符串，不修改任何业务逻辑、数据结构、API 契约、云函数参数、数据库字段名。代码中的变量名、函数名、字段名保持不变。

## Testing Decisions

### 测试策略

纯文案替换，无需单元测试或集成测试。验证方式：

1. **静态检查**：对每个涉及文件，`grep` 搜索旧文案，确认已全部替换且无遗漏；`grep` 搜索新文案，确认写入正确。
2. **模板语法安全**：文案中包含 `{{variable}}` 模板语法的地方，确认替换后模板插值符号完整（`{{xxx}}` 成对闭合）。
3. **构建验证**：执行 `npm run build`（或等效的 miniprogram 构建命令）确认无编译错误。
4. **目视抽查**：在微信开发者工具中逐页浏览，确认各状态下的文案显示正确（空状态、错误状态、成功 toast、弹窗）。

### 不做

- 不写单元测试（纯字符串，测试没有行为价值）
- 不做回归测试（无逻辑变更）
- 不做 E2E 测试

## Out of Scope

- 不改任何业务逻辑或数据模型
- 不改变量名、函数名、字段名
- 不改 tabBar 图标、页面布局、交互流程
- 不改 CONTEXT.md 中的领域术语定义
- 不改预设分类名（荤菜、素菜、主食、汤）和预设菜品名
- 不新增或删除页面
- 不做国际化/i18n 准备
