# Flipia MVP

**Label:** ready-for-agent  
**Created:** 2026-07-17  
**Updated:** 2026-07-20 — 根据 17 张已实现 ticket 和 3 份 ADR 更新

---

## Problem Statement

每天决定吃什么是家庭日常高频痛点。用户面对大脑空白、反复纠结、最终随便凑合的困境，缺少一个能快速、有趣地从自己维护的菜品池里随机决策的工具。现有工具要么菜品不可自定义、要么体验无趣、要么数据无法持久化跨设备同步。

## Solution

Flipia 是一款微信小程序，用户以群组（厨房）为数据单元维护菜品池，按分类（荤/素/主食/汤等）管理菜品，支持多套抽取方案在不同场景快速切换，通过卡片翻转动画随机抽取今日菜单。支持单人使用和多人群组共享，抽取结果、历史记录、菜品图片均存储在微信云开发中，多端同步。

2. 作为用户，我希望我的数据跟随微信账号，以便换手机后数据不丢失。
3. 作为新用户，我希望首次打开时自动获得一批预设菜品，以便不用从零开始维护菜品池。

### 菜品池管理

4. 作为用户，我希望查看按分类 tab 排列的菜品列表，以便快速定位某类菜品。
5. 作为用户，我希望在当前分类 tab 下添加新菜品，以便扩充菜品池。
6. 作为用户，我希望为菜品起名（最多 20 字），以便清晰标识每道菜。
7. 作为用户，我希望为菜品上传 1-3 张图片，以便直观展示菜品外观。
8. 作为用户，我希望不上传图片也能正常添加菜品，以便降低录入门槛。
9. 作为用户，我希望编辑已有菜品的名称和图片，以便修正错误信息。
10. 作为用户，我希望删除不再需要的菜品，以便保持菜品池整洁。
11. 作为用户，我希望单独禁用某道菜（不参与抽取），以便临时排除食材买不到的菜而不用删除。
12. 作为用户，我希望重新启用已禁用的菜品，以便恢复其参与抽取。

## Implementation Decisions

- 原生微信小程序 + TypeScript

- 微信开发者工具开通云环境
- Vitest（Node 环境）测试纯函数模块

### 群组架构（一切皆群组 — ADR-0002）

不存在"个人数据"和"群组数据"两个层次。数据永远归属于群组——单用户场景即成员数为 1 的默认群组（"我的厨房"）。用户可以创建/加入多个群组（上限 5 人/群组），通过顶部群组切换器选择当前活跃群组。

- **权限模型**：群主（创建者）独占踢人/解散权限。所有成员均可增删改数据、邀请新人、修改群组名、退出群组。
- **退出规则**：成员退出后群组数据不带走。仅剩一人时，退出操作自动变为解散（数据销毁）。
- **创建导入**：新建群组时可选择从已有群组复制分类和菜品（复制语义，独立演进），不导入抽取方案和历史。群组名在用户级唯一。
- **成员导入**：已在群组中的成员，可从自己的其他群组导入菜品到当前群组（复制语义），`creatorId` 保留原创建者，同名菜品不去重，`enabled` 状态与源群组一致。
- **预设模板**：新建群组时自动预置荤、素、主食、汤四个默认分类和一套默认抽取方案。所有分类均可自由增删改名。

### 群组切换器（顶部导航栏）

所有 4 个 tab 页面顶部固定显示当前群组名（≤ 8 字，超出省略）+ 切换入口（多群组时显示 ▾）。点击弹出群组列表 + "新建群组"入口。选择后所有页面数据切换到该群组，活跃群组持久化到本地存储。

### 数据库集合

**`groups`**

```
{
  _id: string,           // groupId
  _openid: string,       // 群主（创建者），用于踢人/解散权限校验
  name: string,          // 默认"我的厨房"，组内成员均可修改，用户级唯一
  members: [
    { openid: string, joinedAt: Date }
  ],
  createdAt: Date
}
```

所有成员平权（无 role 字段）。群主身份由 `_openid` 判断。

**`user_config`**

```
{
  _id: string,
  _openid: string,
  groupId: string,
  categories: [
    { id: string, name: string, order: number }
  ],
  drawConfigGroups: [
    {
      id: string,
      name: string,            // 最多 100 字，不重复
      entries: [
        { categoryId: string, categoryName: string, count: number }  // count: 1-5
      ]
    }
  ],
  createdAt: Date,
  updatedAt: Date
}
```

`activeDrawConfigGroupId` 和 `lastDrawnGroupId` 存储在本地存储（`wx.setStorageSync`），不写入云端 `user_config`。

**`dishes`**

```
{
  _id: string,
  _openid: string,
  groupId: string,
  name: string,          // 最多 20 字
  categoryId: string,
  creatorId: string,     // 添加该菜品的成员 openid，仅用于展示（大厨: XXX），非权限控制
  images: string[],      // 云存储 fileID，0-3 条
  enabled: boolean,
  createdAt: Date
}
```

**`draw_history`**

```
{
  _id: string,
  _openid: string,
  groupId: string,
  drawerId: string,          // 执行抽取的成员 openid，用于标注谁抽的
  status: 'active' | 'archived',  // active: 7天内；archived: 软删除
  results: [
    {
      categoryId: string,
      categoryName: string,   // 冗余，防分类删除后历史异常
      dishes: [
        { dishId: string, dishName: string, imageUrl: string }
      ]
    }
  ],
  images: string[],           // 本餐实拍照片，0-3 张
  confirmedAt: Date           // 用户点击「就吃这个」的时间
}
```

### 数据库权限规则

所有集合查询/写入时校验当前用户 openid 是否在当前活跃群组的 `members` 列表中。`_openid` 字段统一存储操作人 openid（用于云开发基础权限），实际权限由群组成员列表决定。

### 初始化流程

用户首次打开 → `app.ts` 中静默登录获取 openid（云函数 login）→ 写入全局状态 → 查询 `groups` 集合判断该 openid 是否已有群组 → 不存在则：

1. 创建 `groups` 文档（单成员，群主 = 当前 openid，名称"我的厨房"）
2. 创建 `user_config` 文档（四个默认分类荤菜/素菜/主食/汤 + 一个默认抽取方案"默认方案"，各分类 count=1）
3. 批量写入预设菜品到 `dishes`（各分类各 5 道常见家常菜）
4. 活跃群组 ID 写入本地存储

已存在群组则直接进入上次活跃群组。初始化全程显示加载态。

`app.ts` 提供 `whenReady()` promise，防止页面在初始化完成前读取 `globalData`。

### Draw Engine（核心抽取模块 — ticket 02）

纯函数，不依赖云 SDK 和小程序运行时，在 Node 环境独立测试。

```
drawDishes(pool: Dish[], config: DrawConfigEntry[]): DrawResult[]
```

- 按分类分组后在组内执行 Fisher-Yates shuffle 取前 N 个
- 同一分类内不出现重复菜品
- 若某分类可用菜品数 < 配置数量，取实际数量（防御处理，不抛错）

```
validateDrawConfig(pool: Dish[], config: DrawConfigEntry[]): { valid: boolean; reason?: string }
```

- 遍历 entries 校验每个分类的启用菜品数是否满足配置数量
- 任一不满足 → `valid: false` + 原因文案（"${分类名}可用菜品不足，需至少 ${count} 道"）

### 方案优先级函数

`resolveEffectiveGroupId(groups: DrawConfigGroup[], activeId?: string, lastDrawnId?: string): string`

优先级：手动设置的生效方案 → 上次抽取的方案 → 列表第一个方案。`activeId` 和 `lastDrawnId` 存储在本地。

### 抽取方案管理 UI（ticket 07 — ADR-0001）

方案列表页：方角网格卡片（一行 3 个），浅绿底色 + 正绿边框。每张卡片显示方案名（最多 2 行）、分类计数 badge。生效方案标注"生效中"，非生效方案显示"设为当前"。点击卡片弹出底部 Modal 编辑。

Modal 内编辑：方案名称输入（最多 100 字）、entries 列表（分类名 + −/＋ 步进器 1-5）、新增 entry（ActionSheet 选择未加入的分类）、移除 entry（至少保留一条）。Modal 内编辑仅本地暂存，“确认”后一次性写入云端。

约束：方案上限 10 套，至少保留 1 套。同名方案禁止（唯一校验）。

### 抽取功能交互（ticket 08）

**卡片机制：**

- 背面：绿色渐变 + "?" + 分类名
- 正面：浅绿底色 + 正绿边框，菜品名 + 第一张图片
- CSS 3D 翻转动画（`rotateY(180deg)` + `backface-visibility`）

**底部操作栏（fixed，`space-between`）：**

- 左侧「返回」— 弹窗确认后退出（无论是否有卡片已翻开）
- 中间「重抽」— 至少一张卡片翻开后启用，按当前方案重新抽取全部
- 右侧「就吃这个」— 所有卡片翻开后激活，写入 `draw_history`

**一键翻牌：** 卡片区域上方 outlined 按钮，点击后按顺序逐张翻转（400ms 间隔），翻牌期间按钮禁用。

**换一道：** 已翻转卡片右下角按钮，卡片先翻回背面，再在同分类内随机替换（排除该分类内所有已分配卡片，含未翻开），再手动翻转揭晓。

**方案快捷切换：** 首页 idle 区域顶部显示当前方案名 + `›`，点击弹出 ActionSheet 从 `drawConfigGroups` 列表切换，即刻写入本地存储并重新校验。切换不展示 loading 态。

### 抽取确认流程

所有卡片翻开后「就吃这个」按钮激活 → 用户点击确认 → 写入 `draw_history`（status: 'active', `drawerId`: 当前用户 openid）→ 检查并将 7 天滚动窗口外的记录软删除（status: 'archived'）→ 页面回到待机状态，支持再次发起抽取。每次确认独立追加记录，支持一日多次。

### 页面布局（ADR-0003）

所有页面禁止使用 `100vh`。`.page` 根容器统一使用绝对定位：

| 页面类型                   | CSS 模式                                                                         |
| -------------------------- | -------------------------------------------------------------------------------- |
| 有 group-switcher + Tab 页 | `position: absolute; top: 88rpx; left: 0; right: 0; bottom: 0; overflow-y: auto` |
| 无 group-switcher 的子页   | `position: absolute; top: 0; left: 0; right: 0; bottom: 0; overflow-y: auto`     |

内部占位元素使用 `min-height: 100%`（相对已定高的 `.page` 容器）。

### 页面结构

4 个 tab：首页 / 菜品池 / 历史 / 我的。所有页面顶部固定显示 `group-switcher` 组件。

- **首页**：「开始抽取」按钮（校验通过时启用）+ 方案快捷切换入口 + 卡片翻转交互区 + 底部操作栏。常态展示今日最新一条已确认结果及今日条数摘要（无记录时显示引导占位），点击摘要跳转历史页。
- **菜品池**：分类横向 tab 切换，当前分类菜品列表按 `createdAt` 倒序，禁用菜品置灰。"添加菜品"入口（浮动按钮），点击菜品卡片进入编辑表单。群组成员数 > 1 时每道菜展示创建者标签（"大厨：XXX"）。
- **历史**：查询 `draw_history`（`status == 'active'`），按 `confirmedAt` 倒序，按天分组。每条显示确认时间 + 菜品列表，群组成员数 > 1 时标注抽取人。支持上传/查看 1-3 张实拍照片。
- **我的**：微信头像/昵称（加载失败时显示默认占位）。分类配置入口、抽取方案入口、群组管理入口（显示当前群组名）。

群组管理子页面（从「我的」页面进入）：

- 群组名（可点击编辑，任意成员可修改）
- 成员列表（头像+昵称，最多 5 人，群主可踢人）
- 邀请按钮（生成微信分享卡片，带 join code）
- 退出群组 / 解散群组按钮（群主不显示退出，非群主不显示解散）

## Testing Decisions

### 主要测试目标：Draw Engine

Draw Engine 是本项目唯一纯函数模块，使用 Vitest 在 Node 环境独立测试，共 52 个测试用例。覆盖：

- **`drawDishes`**: 正常抽取各分类按配置数量返回、同一分类内无重复、菜品不足时取实际数量（防御处理）、空输入返回空数组、多次调用结果不完全相同（随机性验证）
- **`validateDrawConfig`**: 所有分类满足 → `valid: true`；某分类不足 → `valid: false` + 原因文案
- **`resolveEffectiveGroupId`**: 手动设置优先于上次抽取优先于列表第一、无任何设置时取列表第一、空列表时返回空字符串

## Further Notes

- 预设菜品按荤菜/素菜/主食/汤各 5 道常见家常菜预置在 `init-data.ts` 中。
- 图片上传使用微信云存储，fileID 直接存入文档；图片展示使用 `wx.cloud.getTempFileURL` 或直接用 fileID 渲染。
- `draw_history` 中 `categoryName` 和 `dishName` 采用冗余存储，确保分类或菜品被删除后历史记录仍可正常展示。
- `activeDrawConfigGroupId` 和 `lastDrawnGroupId` 存储在本地（`wx.setStorageSync`），不从云端 `user_config` 读写，方案优先级由 `resolveEffectiveGroupId` 纯函数计算。
- 群组名仅在用户级唯一（本人群组列表中不重名），不要求全局唯一。
- 导入菜品时不导入抽取方案——新群组默认创建一套默认方案即可。
- 首页 `onShow` 在同群组内也重新加载 config（修复从子页面返回后方案列表未刷新的问题），但在 drawing 态时跳过以免打断抽取流程。
