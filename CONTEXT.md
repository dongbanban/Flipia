# Flipia

个人/家庭日常菜单随机抽取工具。以群组为数据归属单元，在群组内维护菜品池、按分类组织菜品，通过抽取方案随机决定每餐吃什么。

## Language

**用户 (User)**：
Flipia 的登录个体。拥有用户级数据（插件状态），可属于多个群组并在使用时切换当前活跃群组。用户首次启动即自动创建单人群组。

**群组 (Group) / 厨房**：
数据归属的唯一单元。每个用户至少拥有一个群组，所有菜品、分类、方案、抽取历史均挂在群组下。不存在独立于群组的"个人数据"——单用户场景即成员数为 1 的群组。一个用户可以属于多个群组，在使用时切换当前活跃群组。**厨房**是群组面向用户的展示文案——miniprogram 目录下编写代码时使用 `group`，展示文案均使用"厨房"。
_Avoid_: 家庭、团队、仓库、空间

**菜品 (Dish)**：
群组内维护的每道菜，有名称、所属分类、图片、启用/禁用状态，以及创建者标识（用于展示，非权限控制）。群组内所有成员对菜品有完全操作权限（增删改禁用）。
_Avoid_: 食品、菜式

**创建者 (Dish Creator)**：
菜品在群组内的添加人标识，仅用于展示（如"红烧排骨 大厨：张三"），不赋予特殊权限。
_Avoid_: 作者、添加人

**分类 (Category)**：
菜品归属的类别（如荤菜、素菜、主食、汤），用于组织菜品池和作为抽取方案的 entry slot。新建群组时自动预置四个默认分类作为模板，所有分类均可自由增删改名——不存在不可变的系统分类。
_Avoid_: 类别、分组、标签

**菜品池 (Dish Pool)**：
某个群组内某个分类下的全部菜品集合，是抽取的数据来源。

**导入 (Import)**：
将菜品从用户已有的某个群组复制到另一个群组的操作。导入后两份数据独立演进（复制语义），互不影响。
_Avoid_: 迁移、合并、同步

**抽取方案 (Draw Config Group)**：
一套抽签规则，定义参与抽取的分类及每类抽取数量（1-5 道）。用户可在每个群组内保存多套方案（最多 10 套），不同场景切换不同方案。
_Avoid_: 配置组、抽签模板、抽取配置

**生效方案 (Active Draw Config Group)**：
当前群组内标记为"使用中"的抽取方案。首页抽取按钮始终读取生效方案的规则。

**抽取 (Draw)**：
按生效方案的规则从菜品池中随机选取菜品的动作，使用 Fisher-Yates 洗牌。

**抽取结果 (Draw Result)**：
一次抽取的输出，由每分类的菜品列表组成。用户手动确认后写入历史，记录抽取人。

**图标 (Icon)**：
UI 中使用的语义化图标，通过 `<app-icon>` 组件以语义名引用（如 `CLOSE`、`ADD`、`SEARCH`）。底层使用 `@mp-svg-icons/wechat`（TDesign SVG 图标库），语义名与 TDesign 原生名的映射由 `icon-config` 维护。调用方不感知后端实现。
_Avoid_: 图标名、icon font、glyph

**历史记录 (Draw History)**：
已确认的抽取结果，保留 7 天滚动窗口，超期记录软删除（archived）。历史记录按天分组，每条标记抽取人。数据归属于群组，成员退出后不带走。

**解散 (Dissolve)**：
群组生命周期终点。当群组仅剩一人且该成员执行退出操作时，群组及其所有数据被永久销毁。也可由创建者主动解散。不可逆。
_Avoid_: 删除群组、清空

**群主 (Group Owner)**：
群组的创建者。拥有踢人和解散群组的独占权限。其他操作（邀请、退出、增删改数据、修改群组名）对所有成员开放。
_Avoid_: 管理员、群管理员

## Plugin System

**插件 (Plugin)**：
功能扩展单元。内置在小程序代码中的新功能，需用户手动在插件管理页触发解锁。解锁条件为一次性校验，通过后永久解锁，用户可随时启用/禁用。每个插件在云函数 `plugin-manage` 内集中定义——包含唯一标识、名称、描述、解锁条件评估函数 (`assess`)。前端不作为插件注册清单或共享条件逻辑的持有方。
_Avoid_: 扩展、模块、功能开关

**解锁条件评估 (Plugin Assessment)**：
云函数 `plugin-manage` 中每个插件定义的 `assess(stats: UserStats)` 函数，返回 `{ passed: boolean, progressHint: string, current: number, target: number }`。用户点击解锁时，云函数聚合该用户的统计数据（UserStats），运行对应插件的评估函数，判定是否达标。

**用户统计 (UserStats)**：
解锁条件评估的输入参数对象，包含用户跨群组的累计行为统计（`totalDishCount`、`totalDrawCount` 等）。由云函数实时从数据库聚合查询。

**插件状态 (Plugin State)**：
存储在 `user_plugin` 云数据库集合的用户级文档（每个用户一条记录）。每个插件记录两个字段：

- **unlocked**：`boolean`。用户触发解锁且条件达标后设为 `true`，永久保留。
- **enabled**：`boolean`。仅 unlocked 为 `true` 时可切换。用户手动控制。

`user_plugin` 文档在用户首次访问插件管理页时由 `plugin-manage` 云函数懒创建（全 locked + 全 disabled）。不需要 login 云函数介入。

**plugin-manage 云函数**：
插件系统的唯一后端入口。通过 `action` 参数处理三种操作：
- `list` — 返回所有插件定义 + 用户当前状态 + 未解锁插件的评估进度
- `unlock` — 对指定插件运行 `assess` 评估，通过后更新 `user_plugin`
- `toggle` — 切换指定插件的 `enabled` 字段（仅对 `unlocked` 的插件生效）

**插件组件注入**：
启用后的插件通过其 WXML 组件标签直接嵌入目标页面，页面使用 `wx:if` 绑定本地的插件启用状态控制渲染（如 `<nutrition-panel wx:if="{{pluginEnabled.nutrition}}" />`）。无中心化注入器、无前端注册清单。
_Avoid_: 插槽、挂载点、注入器

**插件管理页**：
用户查看和管理插件的专用页面。从 `plugin-manage` 云函数获取数据，展示每个插件的名称、描述、解锁状态/进度、启用/禁用开关。用户在此手动触发解锁或切换开关。

**插件专属页面**：
提供独立交互界面的插件在 `app.json` 的 `pages` 数组中直接声明新页面路径。
