# ADR-0002: 一切皆群组 — 单一数据维度

**Status:** accepted

## Context

Flipia 最初的领域模型中，"群组 (Group)"被定位为面向未来家庭共享的数据归属单元，与"个人数据"并存为两个层次。在家庭共享场景的 grill 中，我们尝试了以下模型：

- **公用锅模型**（菜品归群组，全员权限）vs **拼桌模型**（菜品归个人，群组是联盟）
- **双账户模型**（个人数据 + 群组数据并存，用户切换视图）
- **投影模型**（群组池是多人个人池的合并视图）

最终发现，引入"个人"和"群组"两个数据层次会导致以下问题：
1. 用户需要理解两套数据视图的切换逻辑
2. 加入群组时个人数据的去向产生合并/迁移/引用三种分支语义
3. 退出群组时数据的归属又产生反向分支

## Decision

**只有一种数据归属单元：群组 (Group)。** 不存在独立于群组的"个人数据"。

- 单用户场景 = 成员数为 1 的自动群组（"我的厨房"）
- 用户可创建/加入多个群组，通过顶部切换器选择当前活跃群组
- 所有数据（菜品、分类、方案、历史）挂在当前活跃群组下
- 新建群组时可选择从已有群组**复制**分类和菜品（复制语义，独立演进）
- 成员退出后数据留在群组；仅剩一人时退出 = 解散（数据销毁）

## Considered Options

- **双数据视图（个人 + 群组）** — 用户需要理解两套视图，加入/退出群组的迁移路径复杂，UI 需要额外一层"当前视图"的标识。拒绝。
- **投影模型（群组 = 个人池合并视图）** — 抽取时需实时合并和去重，性能复杂度高，成员退出时菜品归属判定模糊。拒绝。
- **迁移模型（加入群组时个人数据迁移到群组）** — 退出时无法还原，且"我加进去的菜跟我走还是留下"没有一致答案。拒绝。

## Consequences

- 数据库权限从 `_openid == auth.openid` 改为校验 openid 是否在 `group.members` 列表中
- 群组 API 需支持：创建群组（含导入）、邀请加入、退出、解散、踢人、切换活跃群组
- 菜品表新增 `creatorId` 字段（仅展示用，非权限控制）
- 抽取历史表新增 `drawerId` 字段（标注抽取人）
- 用户首次启动即自动创建单人群组，初始化流程与之前一致
